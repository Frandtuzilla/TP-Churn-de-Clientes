## Hallazgos de Severidad Alta

### [ALTO] H-1: `build_pipeline()` no incluye las 3 features derivadas — el handoff sugiere un patrón que entrena con dataset incompleto

**Ubicación:**
- `src/features/pipeline.py:162-167` — definición de `build_pipeline()`.
- `reports/handoff_to_modeler.md:280-292` — sección §13, snippet sugerido para componer con un clasificador.

**Descripción.** `build_pipeline()` devuelve únicamente el `ColumnTransformer`
base (imputación + RobustScaler + OneHotEncoder). Genera **30 columnas**.
Las 3 features derivadas — `Tenure_was_na`, `tenure_bajo_queja`,
`citytier_alto_queja` — se agregan **fuera** del `ColumnTransformer`,
dentro de `run_pipeline()` (líneas 210, 226-227 de `pipeline.py`). Por
diseño, no forman parte del objeto que `build_pipeline()` devuelve.

El handoff (§13) muestra este snippet como alternativa de uso:

```python
clf = Pipeline([
    ("features", build_pipeline()),
    ("model", DecisionTreeClassifier(class_weight="balanced", random_state=42)),
])
# fitear sobre el raw post-split (X_train_raw, y_train) — pipeline.py hace
# el resto de las transformaciones.
```

El comentario "pipeline.py hace el resto de las transformaciones" es
**incorrecto**: ese `clf` entrenaría con 30 features, no con las 33 que
tiene el parquet. El Modeler distraído que use ese patrón en
`GridSearchCV` no va a notar la diferencia hasta el SHAP (3 features
faltarán).

**Por qué es un problema.** Genera dos universos de features:
- **Si el Modeler usa los parquets** (snippet "load parquet" del handoff):
  33 features. Modelo entrenado con las flags `Tenure_was_na` etc.
- **Si el Modeler compone `build_pipeline()`** (snippet "GridSearchCV"
  del handoff): 30 features. Modelo SIN las derivadas.

Las métricas serán distintas. La interpretabilidad (SHAP, feature
importance) también. Y nadie va a saber por qué — el handoff dice que
"pipeline.py hace el resto" cuando en realidad no es así.

**Buena práctica violada.** *API consistency*: la función pública
`build_pipeline()` debería producir el mismo dataset que el módulo
escribe a disco, o el handoff debería decir explícitamente que esa API
NO incluye las derivadas.

**Cómo lo verifiqué.**

```python
from src.features.pipeline import build_pipeline
pre = build_pipeline()
pre.fit(X_tr, y_tr)
out_cols = pre.get_feature_names_out()
# len(out_cols) = 30
# 'Tenure_was_na' in out_cols → False
# 'tenure_bajo_queja' in out_cols → False
# 'citytier_alto_queja' in out_cols → False
```

**Sugerencia de corrección (sin escribir el código).** Dos opciones:
1. **Envolver las features derivadas como `FunctionTransformer`** dentro
   del `ColumnTransformer` (o dentro de un `Pipeline` wrap encima) para
   que `build_pipeline()` sea autosuficiente.
2. **Reescribir el handoff** dejando claro que la única ruta soportada
   es cargar los parquets (`pd.read_parquet`); eliminar o marcar como
   "uso avanzado, requiere agregar features manualmente" el snippet de
   composición.

La opción 2 es más rápida y suficiente para la entrega del 12/06.

## Hallazgos de Severidad Media

### [MEDIO] H-2: Los parquets incluyen `Churn` mezclado con features — riesgo de leakage del target si el Modeler se distrae

**Ubicación:**
- `src/features/pipeline.py:233-234` — `train_df[TARGET] = y_train.values`.
- `data/processed/features_train.parquet` — columna `Churn` presente entre las features.

**Descripción.** El parquet contiene 34 columnas: 33 features + `Churn`.
El handoff §13 muestra el uso correcto:

```python
y_train = train.pop("Churn")
y_test  = test.pop("Churn")
X_train, X_test = train, test
```

Pero si el Modeler hace `X_train = pd.read_parquet(...)` sin el `.pop()`,
está pasando `Churn` como una feature más → correlación 1.0 con el
target → modelo con accuracy 100% que parece perfecto pero está
entrenando sobre el target.

**Por qué es un problema.** El parquet no es self-protecting: el contrato
con el Modeler depende de que lea el handoff y haga el `.pop()`
correctamente. En una entrega bajo presión de tiempo, ese tipo de
descuido es realista.

**Buena práctica violada.** *Fail-safe defaults*: los artefactos
deberían ser difíciles de usar mal. La práctica habitual es exportar
`X_train.parquet` + `y_train.parquet` por separado, o devolver `(X, y)`
como tupla desde `load_processed()`.

**Sugerencia de corrección.** Dos opciones (cualquiera resuelve):
1. Exportar 4 parquets: `X_train`, `y_train`, `X_test`, `y_test`.
2. Mantener los 2 parquets actuales pero agregar una función helper
   `load_processed_split() -> (X_train, y_train, X_test, y_test)` en
   `pipeline.py` que haga el `.pop()` internamente. El handoff
   recomendaría esa función como entrypoint único.

Distinguir las dos cosas es lo que separa el análisis serio del
análisis "p-hacker".
